Data has never been more valuable than nowadays, thus garantying efficient and robust malware detection algorithms is a major need. Malware recognition modules decide if the execution of a program is a threat or not, based on how it behaves. That is why Huawei has focus on malware detection for several years.

Huawei is very familiar with machine learning research and with transfering ML technics to it's products (smartphones, base stations, data centers, ...). Malware classificaion make no exception: we use machine learning to augment malware detection using various kinds of data.

Today, you have the opportunity to test your machine learning knowledge with a real world (cleaned) data provided by Huawei !

Do your best to provide the most accurate malware detector and enjoy your 24 hours hackathon !

Dynamic Malware Analysis

Malware detection (= classifying a piece of software into malicious or not) is without doubt essential for a secure network environment. The traditional methods of malware detection based on inspecting the software code have been counter-measured by camouflage techniques, such as code polymorphism and metamorphism. To this end, we are interested in the “dynamic malware analysis” approach which monitors and records the software behaviors in real-time operation within a sandbox environment.

Specifically, with the techniques provided by the Enterprise Business Unit of Huawei, our malware detector has the following capabilities. For each examined software:

  1. It records the process generation tree of the software. The main software process can create subprocesses at runtime, which may conduct malicious behaviors (or not).
    The process generation tree of a software is saved in a .txt file:

    • the name of the .txt file is with format ‘*_process_generation.txt’
    • each row is of the form ‘parent_process -> child_process’
    • the row ‘stub.exe_* -> sample.exe_*‘ refers to the sandbox environment initiating the software ‘sample.exe_*‘
    • the order of the rows are random

  2. It records the sequence of behaviors (syscalls), in the order of time, in (Process, RIP, API) triples, where:

    1. Process: the subprocess id where the behavior takes place.
    2. RIP: the API-call returning pointer, i.e., the address of the code which conducts the API function call behavior. With the help of RIP information, we can draw a behavior graph of the software which may vary from very simple to very complicated.
    3. API: the type of the behavior, such as CreateRemoteThread, EnumProcesses, GetForegroundWindow, etc., which may be indicative of whether the software is hostile. In the dataset, they are represented by an index instead of the real name.

    The behavior sequence of a software is saved in a .txt file:

    • the name of the .txt file is with format ‘*_behavior_sequence.txt’
    • each row is a triple of the form ‘process,rip,api’
    • the rows are time-ordered

We have in our disposal a large dataset of classified software, i.e. each dataset entry is a software for which we know if it is malware or not, and we have recorded all its behaviors. Our challenge is to utilize the dataset to build an accurate malware classifier. The dataset is described in details below.

Data

Before getting into the details:

  • There are 200000 training samples, 40000 validation samples, and 32490 test samples.
  • For each sample, there are two txt files, one named ‘*_process_generation.txt’, and the other named ‘*_behavior_sequence.txt’.

The folder with data:

In the folder, you have:

  • ‘true_labels_training.txt’, the true labels for training samples, which contains a sequence of ‘0’s and ‘1’s, in the same order as the indices of the training samples, with ‘1’ referring to malwares and ‘0’ referring to non-malwares
  • training set: (36.3GB) in the ‘training_dataset’ folder,
    • ‘training_index_process_generation.txt’ and ‘training_index_behavior_sequence.txt’ for ‘index’ from 0 to 199999, with 6 digits, 0 padded on the left
    • example: the training sample indexed 1109 corresponds two files, ‘training_001109_process_generation.txt’ and ‘training_001109_behavior_sequence.txt’, and the label for the sample is the 1109th digit (counting from 0) in ‘true_labels_training.txt’
    • both training data and labels will be accessible at the beginning of the hackathon
  • validation set: (7.2GB) in the ‘validation_dataset’ folder,
    • ‘validation_index_process_generation.txt’ and ‘validation_index_behavior_sequence.txt’ for ‘index’ from 0 to 39999, with 6 digits, 0 padded on the left
    • the validation data without labels will be accessible at the beginning of the hackathon
  • test set: (5.8GB) in the ‘test_dataset’ folder,
    • ‘test_index_process_generation.txt’ and ‘test_index_behavior_sequence.txt’ for ‘index’ from 0 to 32489, with 6 digits, 0 padded on the left
    • the test data without labels will be accessible in the 3rd phase of the hackathon
  • evaluation function
    In order to avoid meaningless time waste, we also provide the evaluation function we use.
    • evaluate_exactly_the_same_as_on_the_server.py
      The python evaluation function which we use on the server.
    • evaluate_for_local_test.py
      The python evaluation function which will give the same results as the previous one, but suitable for local test.
      Input the paths of your own truth file and answer file, the roc_auc_score will be printed.

Graph Visualization of a Toy Sample

Here is one toy sample. Its process generation information is:

stub.exe_144f5000 -> sample.exe_13e21000

which means no subprocess is spawned by this software.

Its behavior sequence information is:

sample.exe_13e21000,127b7d4,api_3538
sample.exe_13e21000,127b7d4,api_1814
[…120 lines…]
sample.exe_13e21000,12640b6,api_1955
sample.exe_13e21000,12640d2,api_1439

As we described above, RIP is the address of the code which conducts the API function call behavior. The value of the RIPs is, they can reveal the structure of the behaviors from the sequence. When the same RIP appeared again in the behavior sequence, it means the code is in loop.

We draw a directed graph, with RIP as the node id and API as the node attribute. For two adjacent rows in the behavior sequence, which means the second row ran right after the first one, we put a directed edge from the RIP in the first row to the RIP in the second row. The number of times it happens be the weight on the edge. Then we obtain the following graph:

In the graph, the structure of the software is made clear. You can see that there is a loop executed about 8 times.

Note that, for the same RIP, its corresponding API behaviors may differ, such as node 127b7d4 in the example.

Competition Phases

The competition has three phases (Paris Time).

  • In the first phase (21:00 - 00:00), you have access to the training dataset with labels and the validation dataset without labels. But you cannot submit your results and cannot get feedbacks of the performance on the validation dataset.
  • In the second phase (00:00 - 15:00), you can submit your results on the validation set to the server and get feedbacks of the performance. The criterion is AUC-ROC for binary classification. The submission results will also be posted on a leaderboard so that you can see how well other teams are doing. In total you have at most 200 times of submission.
  • In the third phase (15:00 - 15:30), you will get the key to decrypt the test dataset. You can run your well-tuned algorithm on the testset and have 3 opportunities of submission. The final ranking of the hackathon will be determined by the performance on the test set.

Format of the Submission

Your submission should be a .zip file with an answer.txt file containing your predictions. The name of the .zip file doesn’t matter, but the .txt file has to be named ‘answer.txt’.

For each sample in the validation/test set, in the same order, you must predict a probability of the sample being a malware (from 0 not a malware, to 1 definitely a malware). For example,

0.0001
0.9964
0.5585
…etc.

For the 2nd phase, the evaluation will be made on the validation dataset; for the 3rd phase, on the test dataset.
